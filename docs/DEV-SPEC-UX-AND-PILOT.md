## 1. Контекст

### 1.1 Цель продукта

Веб-система **годового планирования ФОТ по позициям** с:
- подгрузкой факта (Excel → импорт),
- отслеживанием план–факт и прогноза на год,
- квартальной корректировкой и версиями.

### 1.2 Решения стейкхолдера (зафиксировано)

| Тема | Решение |
|------|---------|
| Главный пользователь | **C&B** (владелец процесса, импорт, версии, freeze) |
| Тимлид | Планирует **только свою команду**; правки **параллельно** с другими |
| Директор | Планирует прямых подчинённых; **freeze** = запрет корректировок ниже |
| Viewer | Не нужен в пилоте |
| Факт | **Excel**, один владелец (C&B); ФОТ = все начисления, укрупнённо по статьям |
| Позиции | Глобальная нумерация (П001…); **новые позиции** — только в планировщике |
| План–факт | Все сценарии важны; Δ — **только констатация**, без workflow «исправить план» |
| «Двое на стуле» | Редко; достаточно **индикатора** |
| Прогноз | **Нужен**: факт YTD + план до конца года |
| IN_LIMIT / OVER_LIMIT | **Не бюджетный лимит**, а признак **старой / новой** позиции; для старых — ограничение роста по % (правило — отдельная итерация) |
| Kaiten | Упрощённый UI, без API в пилоте |
| BI | Не делаем |
| Пилот | Несколько тимлидов и юнит-лидов, **500+ позиций**, цикл на **демо-данных** |
| Метрики успеха | Пока не формализованы — качественное тестирование |
| Техпорядок | **Боевой импорт → user testing → PG** |

### 1.3 Проблема текущего UI

Функционал достаточный, но интерфейс воспринимается как **перегруженный**:
- сайдбар совмещает навигацию, dev-контроли и статусы;
- панель «Данные» перекрывает весь экран;
- на `/planning` — два ряда вкладок, несколько баннеров, дубли фильтров;
- технические термины в лицо пользователю (`IN_LIMIT`, `employee_id`, JSON, `v1 DRAFT`);
- `index.css` ~4600 строк без единой дизайн-системы.

**Задача:** сохранить все нужные функции, убрать шум, выстроить понятный путь по ролям.

### 1.4 Жёсткие продуктовые правила (не нарушать при рефакторинге UI)

См. `AGENTS.md` и `PRODUCT-MODEL.md`: факт не правит план; план–факт vs утверждённая версия; сокращение = Closed с M; декрет только FROM_LIST | VACANCY; корректировка только с M<sub>open</sub>.


**Цель:** тимлид за 5 секунд понимает, где он и что можно делать.

**Сейчас:** `AppLayout.tsx` — версия, 4–6 hint-строк, роль MVP, freeze, viewMode, футер про IN_LIMIT.

- Freeze активен: «Правки закрыты директором до …» (без даты — просто факт).

**Критерии приёмки:**
- [ ] Тимлид после переключения роли оказывается на таблице позиций, не на Обзоре с графиками.

## 7. Эпик DATA-1 — Боевой импорт факта (CSV)

**Цель:** C&B загружает **Excel, сохранённый как CSV**, без JSON и без PG.

### 7.1 Формат файла

Базовый шаблон (одна строка = одна статья или агрегат):

| Колонка | Обязательность | Примечание |
|---------|----------------|------------|
| `employee_id` | да | Ключ сотрудника |
| `position_id` | нет | Посадка на стул |
| `month` | да | 1–12 |
| `article` | нет | `BASE`, `BONUS`, `BONUS_PLAN` или код статьи; default BASE |
| `amount` | да* | Сумма по статье |
| `fact_base`, `fact_bonus` | альтернатива | Как в текущем JSON (уже в `factImport.ts`) |
| `tariff_salary` | нет | Тарифный оклад для режима просмотра |

\* Либо пара `fact_base` / `fact_bonus`.

**Расширение (если в Excel несколько статей):** несколько строк с одним `employee_id` + `month`, разный `article` — парсер уже поддерживает `article` + `amount` в `parseFactLineRow`.

### 7.2 Реализация

1. Новый модуль `data/factImportCsv.ts`:
   - парсинг CSV (разделитель `;` или `,` — autodetect по первой строке);
   - нормализация заголовков (`employee_id`, `Employee ID`, …);
   - вызов существующей логики `parseLinesArray` / `inspectFactImport` через адаптер JSON-lines.
2. UI в `SettingsPage` → секция «Факт»:
   - `accept=".csv,text/csv"` + опционально `.xlsx` **не** в scope (только CSV);
   - preview: первые 8 строк, счётчики сотрудников/строк/ошибок (как сейчас для JSON);
   - режим replace / merge — сохранить.
3. Отчёт после импорта: «Загружено N сотрудников, M строк, K посадок; предупреждения: …».

### 7.3 Тесты

`factImportCsv.test.ts`: валидный CSV, неверный month, несколько статей, `;` vs `,`, UTF-8 с BOM.

**Критерии приёмки:**
- [ ] C&B загружает CSV из Excel без ручного JSON.
- [ ] После импорта Обзор и `/analytics` показывают факт (не «—»).
- [ ] Ошибки валидации — понятные, с номером строки.
- [ ] `npm test` покрывает парсер.

**Файлы:** `data/factImportCsv.ts`, `data/factImportCsv.test.ts`, `SettingsPage.tsx`, образец `mvp/frontend/public/samples/fact-import.sample.csv`.
