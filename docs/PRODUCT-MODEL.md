## 1. Цель приложения

Единая веб-система вместо Excel и BI для **годового ФОТ по позициям** (стульям) с версиями, квартальными корректировками и прозрачным план–фактом.

### Пять функциональных блоков

| # | Блок | Суть |
|---|------|------|
| **A** | **Планирование** | Всё, что **известно на момент планирования**: пересмотры, наймы, сокращения, увольнения, декреты, переводы, новые позиции. События в версии/черновике → пересчёт плана. |
| **B** | **Подгрузка факта** | Реальность: фактические увольнения/наймы, **кто реально сидел на стуле**, фактическая ЗП. **Только анализ** отклонений от плана; факт **не редактируется** в UI. |
| **C** | **Корректировка** | С месяца **после текущего квартала** — тот же функционал, что планирование: события, черновик, сравнение версий, влияние на лимит; утверждение → новая рабочая версия. |
| **D** | **План–факт** | Сравнение факта с **актуальной утверждённой (скорректированной) версией** плана, не с устаревшим baseline. |
| **E** | **Выгрузка в Kaiten** | Запланированные и **освободившиеся** позиции → заявки на **найм** и заявки на **изменение в ОТиЗ** (отдельная интеграция, поля описать заранее). |

**Зерно времени:** месяц (не день). Срез занятости и сумм — **на конец месяца**.

**Нумерация позиций:** глобальная П001, П002…

## 2. План vs факт — правила сверки

### Факт (строка импорта)

`employee_id`, `position_id`, `month`, `fact_base`, `fact_bonus`, `tariff_salary`  
Режим отображения: **тарифный ОКЛАД** или **весь ФОТ** (уже в MVP).

### Принцип

- **План** меняется только через **планирование / корректировку** (события, версии).
- **Факт** только **показывает** отклонения; **не** «чинит» план и **не** порождает обязательных событий из-за расхождений.

### Типовые ситуации

| Ситуация | План | Факт | UI |
|----------|------|------|-----|
| **Двое на стуле** | один сотрудник (или переходное состояние) | две строки, сумма может быть выше плана | **Только отклонение** (перерасход / двойная занятость). План **не** меняем автоматически и **не** требуем «догнать» фактом. |
| **Сокращение / закрытие** | с месяца M позиции **нет в плане** (`Closed`, ФОТ = 0) | 0 | Нет «экономии с планом» — плана на слоте уже нет. |
| **Экономия по месяцу** | в плане на конец месяца есть ФОТ на слоте | по факту выплат **0** | Только **Δ план − факт** в отчёте. Это не задача и не «надо внести корректировку» — просто расхождение: в жизни уже иначе, в плане по-прежнему заложено. План меняют только если **сами** решили (планирование/квартальная корректировка), не из-за факта. |
| **Реальный найм** | вакансия / другой ФИО | новый `employee_id` на `position_id` | Кто **реально** пришёл по факту; отклонение по сумме и занятости. |

## 3. Версии, черновик, квартал

| Правило | Решение |
|---------|---------|
| Черновик | **Один** на год; создаёт **C&B**; тимлид видит/правит **только свою team** |
| Окно корректировки | Только с месяца **следующего за текущим кварталом** (пример: Q2 → с **июля**) |
| После утверждения | Новая **версия бюджета**; работаем и сравниваем факт **с ней** |
| Ритм | Начало открытого квартала (июль): утверждение корректировок → **v+1** → план–факт vs **v+1** |
| Индексация | **C&B:** один раз «+X%» (баннер), на весь план; ручные пересмотры **сверху** |
| Согласования | Пока **не блокировать**; маршрут — итерация 2 |
| Консолидация | Итерация 2 (один черновик + срезы) |

**Источник:** Excel — текущий срез; **новые позиции** — в планировщике. **BI не нужен**, если веб стабилен.

## 4. Факт: загрузка

```text
Excel (ежемесячно)  →  импорт  →  БД (monthly_fact_lines + посадка)
                                      │
                    UI: чтение + аналитика отклонений ◄──┘
```

Сравнение: **активная утверждённая версия** (после квартала — v+1) vs факт за те же месяцы.

## 5. Роли

| Роль | Видимость | Правки |
|------|-----------|--------|
| **Тимлид** | Своя **team** | Планирование/корректировка в срезе (если период открыт) |
| **Юнит-лид** | Unit | Срез unit; консолидация — итерация 2 |
| **Директор** | **Департамент** | Чтение по dept; правки по прямым + dept; **закрытие** правок для лидов ниже |
| **C&B** | Всё | Admin: черновик, индексация, версии, факт-импорт, справочники |
| **Viewer** | Scope | Только просмотр |

## 6. Интерфейс (целевой)

### Матрица **позиция × месяц** (ядро)

- **План:** занят / вакансия / **закрыт** (после сокращения плана нет).
- **Факт:** 0 / 1 / 2 сотрудника, отклонение по сумме (без кнопки «исправить план»).
- Клик → drawer: **2 вкладки** — слот/занятость и события+ФОТ; read-only **факт** за месяц.
- **Декрет:** основной на слоте; замещение — **существующий сотрудник** или **вакансия без ФИО** (не ручной «новый сотрудник»).

### Разделы

| Раздел | Блок продукта |
|--------|----------------|
| Планирование (позиции, журнал, согласование) | A |
| Матрица по месяцам | A + B (отклонения) |
| План и факт / Прогноз | D |
| Версии + compare | A, C |
| Корректировка | C (то же UI, другое окно месяцев) |
| Диапазоны, индексация C&B | A |
| Выгрузка Kaiten | E |

## 7. Сценарии по ролям

### Тимлид

| ID | Сценарий |
|----|----------|
| T1 | Планирование team: найм, увольнение, декрет, пересмотр, **сокращение** (план обнуляется с M) |
| T2 | Матрица: увидеть **отклонение** (двое в факте, перерасход) — **без** обязательной правки плана |
| T3 | Матрица: **Δ план − факт** — в плане есть ФОТ, в факте 0; без кнопок «исправить» |
| T4 | План–факт: кто **реально** пришёл на стул по факту vs план |
| T5 | В открытом окне квартала — корректировка с июля (как планирование) |

### C&B

| ID | Сценарий |
|----|----------|
| C1 | Годовое планирование + глобальная индексация |
| C2 | Создать единый черновик; утвердить → **v+1** |
| C3 | Импорт факта; контроль загрузки |
| C4 | План–факт vs **v+1**; влияние на лимит при корректировке |
| C5 | Выгрузка в Kaiten: **найм** (запланированные/новые слоты), **ОТиЗ** (освободившиеся) |

### Директор

| ID | Сценарий |
|----|----------|
| D1 | Обзор dept: план, факт, отклонения, лимит |
| D2 | Закрыть период корректировок для подчинённых лидов |
| D3 | Утверждение версии (с C&B) |

## 8. Kaiten (заготовка полей)

| Тип заявки | Когда | Минимум полей |
|------------|-------|----------------|
| **Найм** | Новая / запланированная вакансия, PLANNED_HIRE | `position_id`, роль, dept/unit/team, лимит, оклад/грейд, месяц выхода |
| **ОТиЗ** | Освобождение слота, сокращение, закрытие | `position_id`, месяц, причина, было/стало по штатке |

## 10. План работ по шагам

### Фаза 1 — Фронт: план, факт, матрица

| Шаг | Задача |
|-----|--------|
| 1 | Вкладка **позиция × месяц**: план (занят/вакансия/**закрыт**) + факт (отклонение, без «исправить») |
| 2 | Факт: N сотрудников на position+month; перерасход — **индикатор**, не workflow |
| 3 | Сокращение: с M **нет плана** на позиции (матрица + события) |
| 4 | Импорт факта: employee, position, month, base, bonus, tariff_salary |
| 5 | План–факт: база сравнения = **активная версия** (подпись в UI) |
| 6 | Окно квартала: правки только с M ≥ начало открытого квартала |
| 7 | Индексация C&B: баннер + применение; не для тимлида |

### Фаза 2 — Роли и корректировка

| 8 | RBAC в UI: team-only, директор freeze, C&B admin |
| 9 | Режим «корректировка» = тот же planning UI, другой allowedMonths |
| 10 | Compare версий + влияние на лимит (уже частично есть) |

### Фаза 3 — БД и Kaiten

| 11 | PG: fact lines + plan versions + events |
| 12 | Роли `users` + scope в API |
| 13 | Выгрузка Kaiten (найм + ОТиЗ), ручной триггер из позиции/журнала |

### Фаза 4 — Процесс

| 14 | Консолидация + согласование (итерация 2) |
| 15 | Приёмка A–E; отказ от BI |
