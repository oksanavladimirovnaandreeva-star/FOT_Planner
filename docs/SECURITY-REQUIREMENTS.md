## Принцип для всей системы

| Правило | Решение |
|---------|---------|
| **Где проверять права** | Только **бэкенд** (API + RLS/политики). UI — подсказки и скрытие кнопок, не защита. |
| **MVP фронт** | Демо-роли в `userAccess.ts` — **не** модель безопасности, а прототип UX до PG/API. |
| **Модель доступа** | **RBAC + атрибуты среза** (роль + `scope_org`), не «чистый» ABAC и не «голый» RBAC. |
| **ПДн** | В БД — шифрование, матрица доступа, аудит; в dev/test — **нет** реальных цифр и ФИО. |

## 4.2 / 8.2 — Авторизация и доступ к данным

### Вопрос ИБ

Достаточно ли RBAC или нужен ABAC?

### Решение для продукта

**RBAC одного уровня недостаточен:** у линейных руководителей может быть **одна роль** (например, `team_lead`), но данные — **разные** (только своя команда / подчинённые).

**Целевая модель: роль + атрибуты среза** (гибрид, не «только атрибуты»):

| Слой | Что задаёт | Примеры |
|------|------------|---------|
| **RBAC** | Класс операций | `plan.edit`, `version.publish`, `fact.import`, `export.run`, `approval.submit` |
| **Атрибуты** | Объём данных | `department_id`, `unit_id`, `team_id`, дерево подчинённости, `plan_version_id` |

Так минимизируются «дыры» (роль явно ограничивает опасные действия) и путаница (срез данных не смешивается с правом «согласовать»).

### Реализация (бэкенд, фаза 3+)

```text
JWT / SSO  →  users.role  +  users.scope_org (или view по оргструктуре)
                    │
                    ▼
         Policy: allow if role_has(permission) AND resource_in_scope(scope, row)
                    │
                    ▼
         PostgreSQL RLS на positions / plan_lines / fact_lines / employees
```

### MVP фронт (сейчас)

- Роли: C&B, директор, юнит-лид, тимлид, viewer + **демо-срез** в `DEMO_ROLE_SCOPE`.
- **Массовая индексация плана** — только C&B (`roleCanApplyMassIndexation` → `cb_admin`); на API — отдельное permission, не делегировать лидам.
- Экспорт снимка плана строится из **отфильтрованного** списка `positions` (видимый срез).
- **Обязательно при API:** дублировать фильтр и запреты на сервере; не доверять `localStorage` роли.

## 5 — Маршруты согласования

| Требование ИБ | Решение |
|---------------|---------|
| Выбор маршрута | Механизм ок; маршрут определяет **система** по правилам + оргструктуре, **не пользователь** |
| Логирование | Каждый переход: кто, когда, версия, шаг, решение; **нельзя** сократить путь вручную |
| Инициирование | Права на старт — по роли (например, тимлид → черновик team, C&B → глобальный черновик) |
| Согласующие | **Назначает система** по оргструктуре (руководитель unit/dept, HR, C&B — из матрицы, не pick-list) |
| Обход шагов | Запрещён: статус `IN_APPROVAL` → только следующий шаг по графу |
| Видимость маршрута | Только участники маршрута + роли с `approval.read` в scope версии |
| Проверки | **Только бэкенд** при `submit` / `approve` / `reject` / `publish` |

### MVP

- UI-маршрут и правила в `planApprovalRules.ts` — **прототип**.
- Кнопки «отправить / опубликовать» в MVP привязаны к роли C&B; в проде — к permission + состоянию версии на API.

### Таблицы аудита (целевые)

- `approval_route_instances`, `approval_step_log` (actor, step, at, comment, ip/session id).

### Целевой API согласования (фаза 3)

| Endpoint | Назначение |
|----------|------------|
| `POST /plan-versions/{id}/submit` | Старт маршрута; система создаёт `approval_route_instance` |
| `POST /plan-versions/{id}/steps/{stepId}/approve` | Текущий шаг; append `approval_step_log` |
| `POST /plan-versions/{id}/steps/{stepId}/reject` | Отклонение с комментарием |
| `GET /plan-versions/{id}/approval-route` | Stepper для UI; только участники + `approval.read` в scope |

Обход шагов и смена маршрута пользователем — **запрещены**; публикация версии — только при `route.status = COMPLETED`.

### Импорт оргструктуры (прод)

- CSV: `department`, `unit`, `team` (UTF-8, разделитель `;` или `,`)
- Только C&B; preview + diff перед replace/merge
- Аудит: `org_import_log` (who, when, rows, mode)

## 7.1 — Хранение (своя БД, ПДн)

| Требование | Решение |
|------------|---------|
| ПДн в БД | ФИО, `employee_id`, суммы — **шифрование at rest** (TDE или column-level для чувствительных полей — уточнить с ИБ) |
| Доступ к БД | Матрица: кто (роль DBA / app / readonly BI); **нет** свободного входа по логину-паролю в обход SSO |
| Креды | Только **Vault** (ротация, без значений в коде/`.env` в репо) |
| Приложение | Подключение сервисной УЗ с минимальными правами; RLS от имени пользователя через `SET app.user_id` или аналог |

## 8.1 — Аутентификация

| Тема | Решение |
|------|---------|
| **MFA** | **Да**, рекомендуется (корпоративный IdP / SSO с обязательным 2FA). В MVP нет — зафиксировать в backlog инфраструктуры. |
| **Break-glass** | **Нет** (ИБ: воспринимается как бэкдор). Аварии — через процесс DBA/владельца продукта с аудитом, не «скрытый» вход в приложение. |
| Сессии | Короткий TTL, привязка к SSO, logout при смене роли в IdP. |

## 9 — Импорт и экспорт

### Экспорт

| Требование | Решение |
|------------|---------|
| Объём | Только то, что **видно на экране** = тот же scope, что и API-лист (фильтры + RLS) |
| Аудит | Каждый экспорт: `user_id`, `at`, `format`, `row_count`, `scope_hash`, `plan_version_id` |
| Формат | CSV (или JSON для интеграций) — **без** исполняемых макросов |

### Импорт

| Требование | Решение |
|------------|---------|
| Формат | **CSV**, не Excel (исключить парсинг XML/макросов xlsx) |
| Кто импортирует | Предложение для согласования с ИБ: |

| Тип данных | Роль / permission | Примечание |
|------------|-------------------|------------|
| Факт (1С / payroll) | `fact.import` → **C&B** (+ сервисная УЗ ETL) | One-way, без правки факта в UI |
| План (массовый) | `plan.import` → **C&B** | С привязкой к `import_batches` + откат |
| Справочники (грейды) | `catalog.import` → **C&B** | Отдельный журнал |
| Лиды | **Нет** массового импорта | Только события в UI в своём scope |

Права импорта — отдельные permission, не «админ всего».

### MVP

- Сейчас: JSON в localStorage, демо-кнопки; факт/план импорт в UI только у C&B.
- При PG: заменить на **CSV API** + `import_batches`; убрать произвольный JSON из прод-UI или ограничить admin+аудит.

## 11 — Размещение и деплой

| Требование | Решение |
|------------|---------|
| Контур | Только **внутренняя сеть**, без публичного доступа |
| Секреты | **Vault**, не код / не plain config |
| Техдоступ vs бизнес | DevOps/админы ОС **без** доступа к расшифрованным ФОТ (маскирование на деплое, ключ шифрования **не** у техроли — см. п. 4.1 в заявке AppSec) |
| Среды | dev/test — синтетика или обезличенный срез; **запрет** копирования prod-дампа «как есть» |
| SoD | Минимум прав, разделение: кто деплоит ≠ кто утверждает бюджет ≠ кто видит все ПДн |

## 12 — Исходный код и CI/CD

| Требование | Решение |
|------------|---------|
| SDLC | Обязательный **code review** + корпоративные **security pipelines** |
| Privileged access | Только разработчики по заявке; prod — через CI/CD, не ручной SSH с полными правами |
| Merge | Вторая пара глаз, **запрет merge без approve** |
| Интеграции (Kaiten, 1С) | Узкий круг, креды в Vault, **лог обращений** |
| Релиз | **Двухконтрольный** (два approver на prod deploy) |
| Техроль и ФОТ | Отдельные УЗ; в тесте нет реальных данных; ключ шифрования недоступен техроли; нет тест-пользователей с prod-правами |

## Сводка: MVP → прод

| Область | MVP (фронт) | Прод (обязательно) |
|---------|-------------|-------------------|
| Доступ | Демо RBAC + scope в JS | RBAC + scope на API + RLS |
| Согласование | UI-правила, кнопки C&B | Серверный граф + audit log |
| Импорт | JSON, C&B в UI | CSV, permissions, `import_batches` |
| Экспорт | JSON среза позиций | CSV + audit, scope = API |
| Auth | Переключатель роли | SSO + MFA |
| ПДн | localStorage | PG + шифрование + Vault |
