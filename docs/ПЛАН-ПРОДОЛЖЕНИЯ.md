## 1. Цель продукта

Веб-система планирования **фонда оплаты труда (ФОТ)** вместо Excel:
- годовой план по календарному году (январь–декабрь);
- позиция = «стул» (бюджетная строка), на одной позиции может быть **несколько сотрудников** (полная ставка каждому, без FTE);
- справочник окладов: **специализация + уровень** → min/mid/max, **CR = оклад / mid**;
- версии плана, события (индексация, найм, правки), пересчёт движком;
- план vs факт; роли **admin** (всё) и **user** (срез оргструктуры).

## 3. Модель данных (связи)

```
org_units (дерево)
positions (external_id: П001, П002… глобально; legacy демо: P-100, P-101)
  ├── hire_status: CARRYOVER | NEW_HIRE | PLANNED_HIRE
  ├── limit_flag: IN_LIMIT | OVER_LIMIT
  └── is_vacancy

employees (external_id: E001 — ключ для факта и плана)
position_assignments (employee ↔ position, spec/level, valid_from/to)

plan_versions (год, label, status)
  ├── dec_snapshots (база декабря прошлого года)
  ├── planned_events (INDEXATION, PLANNED_HIRE, MANUAL_OVERRIDE, …)
  └── monthly_plan_lines (результат recalculate)

monthly_fact_lines (факт по employee_id + month + article)

salary_range_catalog + salary_range_bands (на plan_year)

position_id_sequence — атомарная выдача П001, П002…
users + scope_org_codes
```

**Почему в таблице один ИД позиции у двух сотрудников:** это одна позиция-«стул», два человека — две строки в обзоре; **ИД сотрудника** нужен для план-факт.

## 13. Подтверждённая спецификация (актуально)

### Drawer (обязательное поведение)
- Контекст drawer всегда **по выбранному сотруднику** (не агрегат по позиции).
- Помесячная часть отражает и редактирует: **специализацию**, **уровень**, **оклад (BASE)**, **премию (BONUS_PLAN)**, **итого ФОТ**.
- `CR` может отображаться как вспомогательный показатель, но не является блокирующим для приёмки.

### Сравнение декабрь-к-декабрю
- Показываются:
  - `Дек прошлого года (BASE)`,
  - `Дек плана (BASE)`,
  - `% дек→дек`.
- Правило процента:
  - если `prev > 0`: `(curr - prev) / prev * 100`;
  - если `prev = 0` и `curr > 0`: `100%` (в т.ч. вакансии);
  - если `prev = 0` и `curr = 0`: `0%`.

### Референс дизайна
- Скриншоты BudgetFlow — это ориентир по UX/структуре, **не** пиксель-эталон.

### Правило выполнения задач (обязательное)
- Если требования двусмысленны/неполны:
  1. остановить реализацию,
  2. задать уточняющие вопросы,
  3. получить подтверждение,
  4. только после этого вносить изменения.

   - на загрязненном плане с историческими событиями возможны конфликты старых `MANUAL_OVERRIDE` и новых индексаций;
   - требуется финальная зачистка/нормализация событийного хвоста для выбранного сотрудника/позиции.
3. **Увольнение с сохранением вакансии (IN_LIMIT):**
   - бизнес-ожидание: позиция должна оставаться бюджетированной вакансией, а не уходить в `0`;
   - код доработан, но финальную приемку нужно выполнить на чистом стабильном API-процессе.
4. **Drawer дизайн:**
   - структура и часть UX поправлены, но визуально (типографика/отступы/цвета) еще не доведено до эталонного шаблона.

### Подтвержденные бизнес-правила (актуально)
1. Индексация применяется ко всем активным строкам с месяца события.
2. Если после индексации вручную поменяли оклад и продлили до конца года, следующие индексации должны считаться от этого нового значения.
3. Смена грейда в месяце `M` меняет `spec/level`, пересчитывает `CR` и тянется на последующие месяцы до следующего изменения.
4. Переход сотрудника в другую позицию/департамент:
   - старая позиция освобождается,
   - новая позиция получает сотрудника с месяца перехода.
5. Формат позиции — только `ПNNN` (жестко).

### Технический контекст запуска (критично)
- Базово: API `8000`, UI `5180`.
- На машине наблюдались фантомные listener-процессы на `8000`.
- Для диагностики допустим обход:
  - API `8010`,
  - web с `VITE_API_TARGET=http://127.0.0.1:8010`.
- В каждом отчете/проверке явно фиксировать активный API-порт.

### Минимальный acceptance checklist для нового чата
1. На одном и том же сотруднике:
   - индекс с апреля,
   - ручная правка в мае + продление до конца года,
   - второй индекс с сентября,
   - результат сентября считается от майской ручной базы.
2. Увольнение «в вакансию» не обнуляет бюджет позиции (IN_LIMIT).
3. В drawer есть список изменений и сценарий перехода.
4. Дизайн drawer/page сверен с дефолтным дизайн-шаблоном (см. секцию 17).
